# 인스타 맞팔 탐지기 방향성, 배포, 수익화 계획

상태: 사용자 리뷰용 초안
작성일: 2026-06-10

## 배경

현재 제품은 Toollyst와 겹치는 지점이 많습니다. Instagram 내보내기 안내, 파일 업로드, 언팔로워 결과 테이블, 담백한 다크 결과 화면, Instagram/Threads 프레임, 정렬 버튼, 광고 기반 유틸리티라는 방향이 거의 같습니다.

따라서 벤치마킹의 핵심은 기능 복제가 아니라 톤입니다. 과한 장식보다 빠른 이해, 담백한 결과, Instagram을 쓰는 맥락과 가까운 UI 밀도를 가져오되, 같은 제품처럼 보이면 안 됩니다.

첫 사용자는 대시보드를 보러 오는 것이 아니라 "누가 나를 맞팔하지 않는지" 확인하러 옵니다. 그래서 첫 경험은 언팔로워 확인에 충실해야 합니다. 차별점은 그 다음에 선택적으로 열리는 관계 시각화, export 진단, 다른 SNS export로 확장 가능한 구조에서 가져갑니다.

## 확정한 방향

- 첫 포지션: 관계 분석으로 확장 가능한 Instagram 언팔로워 확인 도구.
- 장기 포지션: 로컬 기반 SNS export 분석 워크스페이스.
- 결과 화면 방향: A안, 언팔로워 우선 + 관계 분석 펼치기.
- 첫 시각화 범위: 관계 분포와 팔로우 기간 분포.
- 문구 톤: 일단 담백한 도구 톤, 이후 더 자연스러운 분석 톤으로 개선.
- 디자인 벤치마크: Toollyst의 담백함과 Instagram스러운 밀도만 참고하고 동일해 보이지 않게 설계.
- 수익화 방향: 디자인과 정확도 안정화 후 작은 광고 실험.
- 배포 방향: Cloudflare Pages 1순위, Vercel 2순위, GitHub Pages 3순위.
- 디자인 구현은 `getdesign.md` 작성 후 시작.

## 제품 구조

결과 화면은 가장 먼저 한 질문에 답해야 합니다.

> 내가 팔로우하지만 나를 팔로우하지 않는 계정이 누구인가?

권장 결과 순서:

1. 요약 카운트와 분석 기준 안내.
2. 언팔로워 테이블: 검색, 정렬, 복사, CSV.
3. export 시점 차이 또는 누락 가능성에 대한 짧은 진단 문구.
4. 선택형 `관계 분석 보기` 영역.
5. 관계 분포 차트.
6. 팔로우 기간 분포 차트.
7. 사용 파일과 파서 진단 정보.

권장 문구:

- `언팔로워 후보`
- `내가 팔로우하지만 상대가 나를 팔로우하지 않는 계정입니다.`
- `관계 분석 보기`
- `팔로우/팔로워/맞팔 비율`
- `팔로우한 지 오래된 계정`
- `분석 기준은 Instagram에서 내보내기 파일을 만든 시점입니다.`

피해야 할 문구:

- 너무 감정적인 문구.
- 상대를 비난하는 문구.
- 실시간 감시처럼 느껴지는 문구.
- Instagram 계정에 직접 접근하는 것처럼 오해될 수 있는 문구.

## 시각화 범위

1차 시각화는 작게 시작합니다.

- 관계 분포: 팔로우, 팔로워, 맞팔, 언팔로워, 언팔로우, 차단, 제한.
- 팔로우 기간 분포: 최근, 3-6개월, 6-12개월, 1-2년, 2년 이상.

시각화는 차별점이지만, 처음부터 대시보드가 화면을 이기면 안 됩니다. 표가 먼저고, 분석은 사용자가 원할 때 펼쳐지는 구조가 맞습니다.

## 배포 계획

첫 배포 추천은 Cloudflare Pages입니다.

이유:

- 현재 앱은 Vite 기반 정적 웹앱입니다.
- Vite 기본 빌드 결과물은 `dist`입니다.
- Cloudflare Pages는 Git 연결 배포와 preview deployment를 지원합니다.
- 초기 정적 유틸리티를 저비용으로 운영하기 좋습니다.
- 광고, 개인정보 페이지, SEO 페이지를 실험하기에 구조가 단순합니다.

기본 설정:

- Build command: `npm run build`
- Build output directory: `dist`
- 배포 전 로컬 확인: `npm run preview`

대안:

- Vercel: Vite 감지와 preview 배포 경험이 좋지만, 현재처럼 정적 앱이면 Cloudflare보다 꼭 우선일 필요는 없습니다.
- GitHub Pages: 가장 단순하지만, 도메인/preview/성장 운영까지 보면 우선순위는 낮습니다.

공개 배포 전에 필요한 것:

- 도메인 또는 안정적인 서브도메인.
- 사이트 이름과 메타 설명.
- 개인정보 처리 안내.
- `Instagram/Meta 공식 서비스가 아님` 안내.
- `파일은 이 앱의 서버로 전송되지 않고 브라우저에서 분석됨` 안내.
- 광고 도입 시 광고/쿠키 안내.
- `robots.txt`.
- `sitemap.xml`.
- Open Graph 이미지.
- Instagram export 다운로드 가이드.
- 오류/진단/결과 해석 페이지 또는 섹션.
- 데스크톱/모바일 배포 후 업로드 smoke test.

## 수익화 계획

목표는 거창한 매출보다 먼저 실제로 1,000원이라도 벌 수 있는 구조를 검증하는 것입니다. 다만 수익화가 신뢰를 해치면 이 제품의 핵심 장점이 사라집니다.

권장 순서:

1. 결과 디자인을 정리하고 언팔로워 확인 흐름을 빠르게 만든다.
2. 최소 차별점인 관계 분포와 팔로우 기간 시각화를 붙인다.
3. 개인정보/공식 서비스 아님/가이드/오류 설명을 포함해 배포한다.
4. 실제 광고 스크립트 전에 광고 자리만 먼저 잡는다.
5. 레이아웃, 성능, 신뢰감 저하 여부를 확인한다.
6. Google AdSense와 Kakao AdFit을 단계적으로 실험한다.
7. 핵심 레이아웃이 안정된 뒤 데스크톱 우측 보조 광고 영역을 추가한다.

허용할 광고 위치:

- 가이드 상단 또는 하단.
- 결과 화면 하단.
- 데스크톱 전용 우측 보조 영역.

피해야 할 광고 위치:

- 파일 업로드 영역.
- 분석 버튼 주변.
- 오류와 진단 메시지 내부.
- 결과 테이블 행 사이.
- 요약 결과와 첫 언팔로워 테이블 사이.

광고 심사를 생각하면 순수 SPA 도구 하나만으로는 약할 수 있습니다. export 다운로드 방법, 결과 해석, 개인정보 처리, 오류 해결 같은 콘텐츠 표면을 함께 만들어야 합니다.

## 추가로 신경써야 할 부분

개인정보와 신뢰:

- 업로드 전에 로컬 분석 원칙을 명확히 보여줘야 합니다.
- 광고가 붙으면 제3자 스크립트가 로드된다는 점을 설명해야 합니다.
- 광고 스크립트가 파싱된 파일 내용, 사용자명, 분석 결과에 접근하지 못하게 경계를 분리해야 합니다.

정확도:

- Instagram export 구조가 바뀌는 케이스를 fixture로 계속 쌓아야 합니다.
- 잘못된 카운트, multipart followers 파일, 삭제 계정, 차단/제한 파일, 폴더 업로드 회귀 테스트가 필요합니다.
- 결과가 이상하다는 피드백을 받았을 때 사용 파일과 skip count를 UI에서 확인할 수 있어야 합니다.

SEO와 유입:

- 앱 화면 하나만으로는 검색 유입이 약합니다.
- `인스타그램 내보내기 다운로드 방법`, `언팔로워 결과 해석`, `분석이 틀려 보이는 이유`, `개인정보 안내` 같은 콘텐츠가 필요합니다.
- 우선 한국어 검색 의도에 맞추고, 이후 영어 확장을 검토합니다.

법적/플랫폼 리스크:

- Instagram 또는 Meta 공식 서비스가 아님을 명확히 해야 합니다.
- 실시간 Instagram 조회처럼 보이면 안 됩니다.
- 로그인, 크롤링, private API, 계정 자동화는 하지 않는 방향을 유지합니다.

디자인:

- 첫 viewport는 실제 작업인 업로드/분석으로 바로 이어져야 합니다.
- Toollyst의 담백함은 가져오되, 결과 시각화와 진단으로 다른 제품처럼 보여야 합니다.
- 차트는 주인공이 아니라 2차 정보입니다.
- 디자인 구현 전 `getdesign.md`로 화면 톤을 한 번 더 맞춥니다.

성능:

- 큰 ZIP은 메모리와 렌더링을 흔들 수 있습니다.
- 파일 크기 가드, 진행 상태, 실패 상태를 더 명확히 해야 합니다.
- 모바일 업로드는 데스크톱과 별도로 확인해야 합니다.

분석/측정:

- 분석 이벤트를 넣더라도 개인정보 없는 집계 이벤트만 허용합니다.
- 허용 이벤트: 페이지 조회, 업로드 시작, 분석 성공, 분석 실패, CSV 다운로드.
- 금지 이벤트: 사용자명, 결과 목록, 파일명, 원본 export 내용.

운영:

- 런칭 체크리스트가 필요합니다.
- 배포 후 업로드와 결과 렌더링 smoke test가 필요합니다.
- 실제 광고 스크립트 전에 광고 placeholder 레이아웃부터 검증합니다.
- `.superpowers/` 같은 브레인스토밍 산출물은 git에 넣지 않습니다.

## 남은 질문

- 최종 사이트 이름과 도메인.
- Cloudflare Pages로 확정할지, Vercel을 쓸지.
- 가이드/오류/개인정보를 별도 페이지로 만들지, 앱 내부 섹션으로 둘지.
- 광고 네트워크 순서: AdSense 먼저, Kakao AdFit 먼저, 또는 단계적으로 둘 다.
- 폰트와 이미지 자산을 배포 전에 self-host할지.
