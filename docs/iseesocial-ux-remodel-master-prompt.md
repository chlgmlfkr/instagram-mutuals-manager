# ISeeSocial UX Remodel Master Prompt

아래 프롬프트는 ISeeSocial 개편 작업을 역할별 스레드와 분업해 진행하기 위한 최종 실행 프롬프트다.  
기존 `README.md`, `DESIGN.md`, 런칭 체크리스트와 충돌하는 부분이 있으면 **사용자가 첨부한 복사 프롬프트를 우선한다**. 특히 기존 문서의 “랜딩 페이지를 피한다”는 방향보다, 이 프롬프트의 **5단계 상태 분리형 UX**와 **첫 화면 설득/CTA 집중** 요구가 우선이다.

---

## 0. 역할과 작업 방식

당신은 **시니어 UX/UI 디자이너**, **서비스 기획자**, **전환율 최적화(CRO) 전문가**, **프론트엔드 UX 리뷰어**, **React/Vite/Tailwind 기반 웹앱 설계자**다.

작업은 역할별 스레드 또는 서브에이전트로 분업한다.

1. **UX/UI 디자인 역할**
   - 첫 화면, 업로드 흐름, 분석 준비 화면, 결과 화면, 모바일/데스크톱 레이아웃을 검토한다.
   - 정보량을 줄이고 CTA를 강하게 만드는 방향을 제안한다.

2. **서비스 기획/CRO 역할**
   - 사용자 행동 흐름, 전환 목표, 상태별 CTA, 카피라이팅, 신뢰 UX를 검토한다.
   - “처음 들어온 사용자가 다음 행동을 5초 안에 이해하는지”를 기준으로 판단한다.

3. **정책/수익화/SEO 역할**
   - 개인정보, 광고 위치, AdSense/Kakao AdFit 심사 리스크, Instagram/Meta 비공식 고지, SEO 콘텐츠를 검토한다.
   - 광고가 로컬 분석 신뢰를 해치지 않는지 확인한다.

4. **프론트엔드 구조/검증 역할**
   - React/Vite/Tailwind 컴포넌트 구조, 상태 관리, 회귀 테스트, 배포 전후 검증 기준을 정리한다.
   - `npm run verify` 통과와 모바일/데스크톱 smoke 기준을 포함한다.

각 역할은 독립적으로 검토한 뒤, 최종 답변 또는 구현 계획에 통합한다. 단, 역할별 의견이 충돌하면 **사용자가 첨부한 복사 프롬프트의 요구사항**을 우선한다.

## 1. 서비스 맥락

서비스명은 **ISeeSocial**이다. 한국어 별칭은 `아시셜`로만 사용하고, 공식 표기는 `ISeeSocial`로 고정한다.

사용자는 Instagram에서 다운로드한 **내보내기 ZIP 파일**을 업로드하고, 브라우저 안에서만 분석하여 **인스타 언팔로워 후보**를 확인한다.

핵심 가치는 다음과 같다.

- **로그인 불필요**
- **서버 업로드 없음**
- **브라우저 로컬 분석**
- **ZIP 파일 우선 지원**
- **CSV 내보내기**
- **개인정보 보호**
- **Instagram 또는 Meta의 공식 서비스 아님**

금지 방향:

- Instagram 로그인, 공식 API, 비공식 private API, 크롤링, 계정 자동화
- 사용자 데이터 저장, cloud sync, 결과 목록 서버 전송
- 팔로워 늘리기, 성장 해킹, 자동 관리 도구처럼 보이는 표현
- 업로드 영역/분석 버튼/결과 테이블 근처 광고

## 2. 핵심 문제 정의

현재 UI의 핵심 문제는 다음 한 문장이다.

**현재 UI는 랜딩, 업로드, 분석 준비, 분석 중, 결과 화면이 한 화면에 섞여 있어 사용자의 다음 행동을 흐리게 만든다.**

현재 화면에는 다음 요소들이 동시에 노출되어 정보량이 과하다.

- 랜딩/광고성 소개 문구
- ZIP 업로드 카드
- 분석 시작 버튼
- 개인정보 안내
- 로컬 분석 설명
- 공식 서비스 아님 안내
- 다운로드 가이드
- 분석 상태 카드
- 결과 미리보기
- 결과 대시보드
- 그래프/관계 분석 영역
- CSV 내보내기 기능

목표는 예쁜 화면을 만드는 것이 아니라 **정보량 감소**, **CTA 집중도 향상**, **신뢰감 강화**, **분석 흐름 명확화**, **결과 해석력 강화**다.

## 3. 최종 UX 방향: 5단계 상태 분리

페이지를 다음 5단계 상태로 명확히 분리한다.

1. **랜딩 화면**
2. **ZIP 업로드 화면**
3. **ZIP 선택 완료 / 분석 준비 화면**
4. **분석 중 화면**
5. **분석 완료 / 결과 화면**

각 단계에서 사용자가 해야 할 행동은 하나만 강하게 보여야 한다.

상태 모델은 다음을 기본으로 한다.

```ts
type AppViewState =
  | 'idle'
  | 'upload'
  | 'fileSelected'
  | 'analyzing'
  | 'success'
  | 'error';
```

필요 데이터 상태:

- `selectedFile`
- `fileName`
- `fileSize`
- `analysisProgress`
- `currentStep`
- `followers`
- `following`
- `unfollowerCandidates`
- `mutuals`
- `notFollowingBack`
- `parseErrors`
- `activeTab`
- `searchQuery`
- `selectedAccounts`
- `exportFormat`
- `errorMessage`

## 4. 랜딩 화면 요구사항

첫 진입 화면은 기능을 모두 보여주는 화면이 아니라, 사용자를 설득하고 첫 행동으로 이끄는 **짧은 랜딩 상태**여야 한다.

단, 과한 마케팅 히어로나 장식형 랜딩은 피한다. 목적은 “서비스 가치 이해 → ZIP 업로드로 이동”이다.

첫 화면에는 아래 요소만 남긴다.

- 서비스 이름
- 강한 헤드라인
- 짧은 설명 문구
- 핵심 신뢰 배지
- 메인 CTA 버튼
- 보조 CTA 버튼

추천 헤드라인:

**인스타 언팔로워 후보, 로그인 없이 확인하세요.**

추천 보조 문구:

**Instagram 내보내기 ZIP을 브라우저 안에서만 분석합니다. 파일은 서버로 전송되지 않습니다.**

추천 신뢰 배지:

- **로그인 불필요**
- **서버 미전송**
- **브라우저 로컬 분석**

추천 CTA:

- 메인 버튼: **ZIP으로 확인 시작하기**
- 보조 버튼: **다운로드 방법 보기**

첫 화면에서 크게 노출하지 말 것:

- ZIP 업로드 카드
- 분석 상태
- 결과 카드
- 개인정보 긴 설명
- 보조 폴더 업로드
- CSV/복사 기능
- 관계 그래프

## 5. ZIP 업로드 화면 요구사항

사용자가 **ZIP으로 확인 시작하기**를 누르거나 아래로 스크롤하면 실제 업로드 화면으로 이동한다.

이 영역의 목적은 하나다.

**Instagram 내보내기 ZIP 파일을 선택하게 만드는 것**

포함 요소:

- **Instagram 내보내기 ZIP 선택**
- ZIP 선택 버튼
- 선택 전 안내 문구
- 다운로드 방법 보기
- 로컬 분석 안내 배지
- ZIP이 없을 때 사용하는 보조 폴더 업로드 옵션

보조 폴더 업로드는 처음부터 크게 보이지 않게 한다.

추천 방식:

- 접힘 카드
- 보조 옵션 버튼
- “ZIP이 없나요?” 형태의 작은 안내

---

## 6. ZIP 선택 완료 / 분석 준비 화면 요구사항

ZIP 파일을 선택한 직후에는 대시보드가 아니라 **분석 준비 완료 화면**을 보여준다.

이 단계에서 사용자가 해야 할 행동은 하나다.

**선택한 ZIP을 확인하고 분석 시작 버튼을 누르는 것**

중앙 메인 영역에 다음 요소만 크게 보여준다.

- **ZIP 선택 완료**
- 선택된 파일명
- 파일 크기
- 짧은 안전 안내 문구
- 큰 **분석 시작** 버튼
- 작은 **다른 ZIP 선택** 버튼
- 작은 **다운로드 방법 보기** 버튼

추천 문구:

```text
ZIP 선택 완료

instagram-data.zip

이 파일은 서버로 전송되지 않고, 브라우저 안에서만 분석됩니다.

[분석 시작]

[다른 ZIP 선택] [다운로드 방법 보기]
```

중요: **분석 시작 버튼은 왼쪽 사이드바가 아니라 중앙 메인 카드 안에 크게 배치**한다.

분석 전에는 아래 요소를 숨기거나 축소한다.

- 팔로워 수 카드
- 팔로우 수 카드
- 언팔로워 후보 수
- 맞팔 수
- 분석 실패 수
- 결과 미리보기
- 분석 메모
- CSV 내보내기
- 관계 그래프
- 긴 개인정보 설명
- 긴 로컬 신뢰 설명

분석 전 안내는 짧게 유지한다.

**ZIP 파일이 준비되었습니다. 분석을 시작하면 팔로워/팔로우 데이터를 비교해 언팔로워 후보를 보여드립니다.**

## 7. 분석 중 화면 요구사항

분석 중에는 반드시 **로딩 애니메이션**과 **진행 단계 표시**가 있어야 한다. 목적은 장식이 아니라 작동 중이라는 확신을 주는 것이다.

추천 진행 단계:

1. **ZIP 파일 확인 중**
2. **Instagram 데이터 구조 읽는 중**
3. **팔로워 목록 분석 중**
4. **팔로우 목록 분석 중**
5. **언팔로워 후보 정리 중**
6. **결과 화면 준비 중**

추천 UI:

- 진행률 바
- 단계별 체크 표시
- skeleton UI
- 버튼 로딩 상태
- 부드러운 카드 전환
- “브라우저에서 분석 중” 안내 문구

추천 문구:

**브라우저에서 ZIP을 분석하는 중입니다.**  
**파일은 서버로 전송되지 않습니다.**

## 8. 분석 완료 / 결과 화면 요구사항

분석 완료 후 화면은 **결과 해석 화면**처럼 보여야 한다.

분석 완료 후 사용자의 관심사는 더 이상 업로드가 아니다. 왼쪽 ZIP 업로드 영역 또는 상단 업로드 영역은 축소한다.

추천 형태:

```text
현재 분석 파일

instagram-data.zip

[다른 ZIP 분석하기]
```

기존의 긴 ZIP 업로드 설명, 보조 폴더 업로드, 분석 시작 버튼은 숨기거나 접힘 처리한다.

결과 화면에서 가장 중요한 정보는 **언팔로워 후보 수**다.

예시:

```text
언팔로워 후보 539명

내가 팔로우하지만 나를 팔로우하지 않는 계정입니다.

전체 팔로잉 854명 중 약 63.1%입니다.

기준: 내가 팔로우하지만 나를 팔로우하지 않는 계정
```

비율을 보여줄 때는 반드시 계산 기준을 함께 표시한다.

결과 화면에서 강조해야 할 정보:

- **언팔로워 후보 수**
- **나를 팔로우함**
- **맞팔**
- **팔로워**
- **팔로잉**
- **읽지 못한 항목**
- **제한/차단**
- **CSV 내보내기**
- **선택한 계정 복사**
- **전체 후보 복사**
- **검색**
- **정렬**
- **필터**

결과 해석 안내를 요약 카드 아래에 작게 배치한다.

**이 결과는 Instagram 내보내기 파일을 기준으로 계산됩니다.**  
실시간 Instagram 상태와 다를 수 있으며, 비공개 계정·삭제된 계정·데이터 누락에 따라 일부 차이가 생길 수 있습니다.

## 9. 문구 개선 기준

전체 문구는 너무 마케팅스럽지 않게, **신뢰감 있고 직관적인 한국어**로 작성한다.

| 기존 문구 | 개선 문구 |
|---|---|
| 관계 분석 보기 | **그래프 보기** |
| 관계 분석 | **관계 그래프** |
| 관계 분포 | **팔로우 관계 비율** |
| 언팔로워 후보 요약 | **분석 결과 요약** |
| Relationship Ledger | **계정 목록** 또는 **분석 결과 목록** |
| Analysis Status | **분석 결과** |
| 해석 실패 | **읽지 못한 항목** |
| 결과 탭 준비 | **결과 정리 완료** |
| 개인정보와 광고 경계 | **개인정보는 어떻게 처리되나요?** |
| Instagram ZIP 선택 | **Instagram 내보내기 ZIP 선택** |
| 내보내기 ZIP 업로드 | **다운로드한 ZIP 그대로 업로드** |
| 먼저 ZIP 또는 폴더를 선택하면 분석 버튼이 활성화됩니다. | **ZIP을 선택하면 분석을 시작할 수 있습니다.** |
| 언팔로워 후보를 먼저 보여줍니다 | **나를 팔로우하지 않는 계정을 먼저 보여줍니다** |

## 10. 그래프 / 데이터 시각화 요구사항

분석 완료 후에는 숫자 카드만 보여주지 말고, 결과를 이해할 수 있는 그래프를 강화한다.

단, 그래프는 결과 테이블과 핵심 액션을 밀어내면 안 된다. 모바일에서는 요약 뒤, 계정 목록 전후의 적절한 위치에 둔다.

### 10.1 팔로우 관계 비율 그래프

목적: 전체 관계 상태를 한눈에 보여준다.

포함 항목:

- 언팔로워 후보
- 나를 팔로우함
- 맞팔
- 팔로워
- 팔로잉
- 제한
- 차단
- 읽지 못한 항목

추천 표현:

- 가로 막대 그래프
- 누적 막대 그래프
- 항목이 적을 경우 도넛 차트

### 10.2 맞팔 / 비맞팔 비교 그래프

목적: 내가 팔로우하는 계정 중 **맞팔**과 **언팔로워 후보**를 비교한다.

표시 항목:

- 맞팔
- 언팔로워 후보
- 기타
- 읽지 못한 항목

추천 표현:

- 비교 막대 그래프
- 비율 카드
- 간단한 도넛 차트

### 10.3 언팔로워 후보 비율 카드

예시:

```text
언팔로워 후보 539명

전체 팔로잉 854명 중 약 63.1%입니다.

기준: 내가 팔로우하지만 나를 팔로우하지 않는 계정
```

### 10.4 탭 기준 그래프

사용자가 탭을 바꾸면 그래프도 해당 탭 기준으로 바뀔 수 있다.

예시:

- 언팔로워 후보 탭: 후보 수, 선택 수, 내보내기 가능 수
- 맞팔 탭: 맞팔 수, 전체 팔로잉 대비 비율
- 나를 팔로우함 탭: 나는 팔로우하지 않지만 나를 팔로우하는 계정 수
- 읽지 못한 항목 탭: 파싱 실패 또는 누락된 항목 수

### 10.5 팔로우 시점 분포 그래프

확실하지 않음: Instagram 내보내기 ZIP 안에 시간 정보가 포함되어 있고 현재 파서에서 읽을 수 있을 때만 적용한다.

가능하다면 다음 그래프를 추가한다.

- 월별 팔로우 수
- 최근 팔로우한 계정 수
- 오래전에 팔로우한 계정 수
- 팔로우 시점이 없는 계정 수

데이터가 없다면 이 그래프는 숨기거나 아래 문구를 보여준다.

**Instagram 내보내기 파일에 시간 정보가 없어 시점 그래프를 표시할 수 없습니다.**

## 13. 애니메이션 전략

사이트가 너무 정적으로 느껴지지 않도록 상태 변화 중심 애니메이션을 설계한다.

화려한 장식이 아니라 **사용자가 상태 변화를 확실히 인지하게 만드는 애니메이션**이어야 한다.

| 상황 | 추천 애니메이션 | 목적 |
|---|---|---|
| 첫 화면 진입 | 헤드라인과 CTA fade-up | 첫인상 강화 |
| CTA 클릭 | 업로드 섹션으로 smooth scroll | 흐름 연결 |
| ZIP 선택 완료 | 파일 카드 확장 + 체크 아이콘 등장 | 선택 완료 인지 |
| 파일명 표시 | fade-in | 상태 변화 인지 |
| 분석 시작 | 버튼 로딩 상태 전환 | 작동 중이라는 확신 |
| 분석 중 | progress bar / stepper | 기다리는 불안 감소 |
| 분석 완료 | 완료 체크 아이콘 등장 | 성공 인지 |
| 숫자 카드 | 0에서 실제 숫자로 count-up | 결과 도출 체감 |
| 그래프 | 막대가 차오르는 애니메이션 | 데이터 시각화 강화 |
| 결과 리스트 | skeleton에서 실제 목록으로 전환 | 로딩 안정감 |
| 탭 전환 | 부드러운 fade 전환 | 화면 변화 인지 |
| 개인정보 안내 열기 | accordion slide-down | 정보량 조절 |

구현은 우선 CSS transition 기반을 선호한다. `Framer Motion` 같은 새 의존성은 꼭 필요할 때만 검토하고, 추가 전 이유를 명확히 설명한다.

`prefers-reduced-motion`을 지원한다.

모바일에서는 여러 카드가 동시에 보이면 복잡하다. 모바일은 단일 컬럼으로 구성한다.

## 20. 최종 답변 형식

작업이 “분석/제안” 요청일 때는 아래 형식으로 답변한다.

1. **전체 평가 요약**
   - 현재 UI 완성도 100점 만점 평가
   - 가장 큰 장점 3개
   - 가장 큰 문제점 3개

2. **핵심 문제 한 문장 정의**
   - 예: **현재 UI는 랜딩, 업로드, 분석 준비, 결과 화면이 한 화면에 섞여 있어 사용자의 다음 행동을 흐리게 만든다.**

3. **우선순위별 개선 목록**

   | 우선순위 | 문제 | 이유 | 개선안 | 기대 효과 |
   |---|---|---|---|---|

4. **새 정보구조 제안**

   | 순서 | 섹션 | 목적 | 포함 요소 | 제거/축소할 요소 |
   |---|---|---|---|---|

5. **상태별 화면 설계**

   | 상태 | 핵심 목적 | 메인 UI | 보조 UI | CTA |
   |---|---|---|---|---|

6. **화면별 상세 피드백**
   - 랜딩 화면
   - ZIP 업로드 화면
   - ZIP 선택 완료 화면
   - 분석 중 화면
   - 분석 완료 화면
   - 결과 리스트 화면
   - 개인정보/신뢰 안내 화면
   - 모바일 화면

7. **카피라이팅 개선안**

   | 기존 문구 | 문제점 | 개선 문구 |
   |---|---|---|

8. **애니메이션 개선안**

   | 상태 변화 | 추천 애니메이션 | 구현 방식 | 기대 효과 |
   |---|---|---|---|

9. **그래프/데이터 시각화 개선안**

   | 그래프 | 목적 | 표시 데이터 | 추천 표현 방식 |
   |---|---|---|---|

10. **React/Vite/Tailwind 컴포넌트 구조**
    - 컴포넌트 트리와 각 컴포넌트 역할

11. **Quick Win 5개**

12. **장기 개선안 5개**

13. **최종 추천 방향**
    - 예: **첫 화면은 설득에 집중하고, ZIP 선택 후에는 분석 시작에 집중하고, 분석 완료 후에만 결과 대시보드를 보여주는 단계형 UX로 재설계해야 한다.**

## 21. 답변 원칙

- 막연하게 “깔끔하다”, “좋다”라고만 말하지 않는다.
- 반드시 **왜 문제인지**, **어떻게 고칠지**, **기대 효과가 무엇인지**를 설명한다.
- 확실하지 않은 부분은 **확실하지 않음**이라고 표시한다.
- 실제 개발자가 바로 반영할 수 있을 정도로 구체적으로 작성한다.
- 중요한 키워드는 **볼드체**로 강조한다.
- 내용이 전환될 때마다 `---` 구분선을 넣는다.
- 기존 문서와 충돌하면 첨부된 복사 프롬프트를 우선한다.
